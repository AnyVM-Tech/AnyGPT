# AGENTS.md

## Repository notes
- `apps/langgraph-control-plane` is the newer LangGraph-based control plane. Its orchestration lives primarily in `src/workflow.ts`, with runner/status wiring in `src/index.ts`.
- The safest AI-provider migration point is the centralized control-plane AI helper path in `workflow.ts` rather than node-by-node graph changes.
- `CONTROL_PLANE_AI_PROVIDER=openhands` enables the OpenHands provider mode. The TypeScript side routes through a Python bridge at `apps/langgraph-control-plane/openhands_bridge.py`.
- `apps/langgraph-control-plane/src/semanticMemoryStore.ts` still calls `${baseUrl}/embeddings` directly and remains OpenAI-compatible today; the OpenHands migration completed here covers the centralized chat/code-edit AI path, not semantic-memory embeddings.

- Current OpenHands bridge prerequisite: `openhands-sdk` must be installed in the Python environment used by the control plane. The repo does not currently declare that dependency in a dedicated Python project file under `apps/langgraph-control-plane`.
- Provider/backend labels should use `describeControlPlaneAiBackend(...)` instead of raw `baseUrl` so status and errors stay accurate across `openai-compatible` and `openhands-sdk` modes.
- `apps/anyscan/index.html` currently owns the operator `/app` findings publication UI. Publication state is held in `state.findingPublications`; keep `loadFindingPublications()`, `upsertFindingPublicationRecord(...)`, `renderFindingPublications(...)`, and the per-finding moderation panel in sync when changing publication or reviewer-notes wiring.
- Important moderation UI nuance: the `/app` publish/suppress handler must always send `public_summary` and `reviewer_notes`, even when the operator clears those fields. Blank strings let the Rust backend clear reviewer notes and restore `default_public_finding_summary(...)`; omitting the fields preserves prior values instead.
- `apps/anyscan/src/dragonfly_store.rs` now has regression coverage for public finding moderation/search behavior: published findings are searchable, suppressed findings are hidden from the public search surface, reviewer notes persist/clear through moderation updates, and blank public summaries fall back to `default_public_finding_summary(...)`. Useful verification command: `cd apps/anyscan && cargo test dragonfly_store::tests:: -- --nocapture`.
- `/app` reviewer notes are private-by-permission, not just hidden in one render branch: non-moderator sessions must never receive `reviewer_notes` from `GET /api/findings/publications` or the `/api/events` SSE `PublicFindingModerated` payload. Keep `index.html` UI gating and `src/bin/anyscan-api.rs` sanitization helpers aligned.
- `/app` findings publication state should update live from SSE moderation events as well as initial fetches. Keep `applyApiEvent(...)`, `upsertFindingPublicationRecord(...)`, `renderFindings(...)`, and `describeEvent(...)` privacy-safe and in sync so operator views refresh immediately without exposing reviewer notes.

- Public findings search UIs should treat empty/null query objects as inactive. In both `/app` and `/`, use helper-based query detection rather than `Object.keys(query).length` because queries often carry null fields.
- `apps/anyscan` has no local SQLite/in-memory store fallback today; `AnyScanStore::from_config(...)` always builds the Dragonfly/Redis client from `StorageConfig`. Manual `/app` verification therefore requires a reachable Redis/Dragonfly instance (default config targets local Redis on 127.0.0.1:6379).
- The public `/` findings browser in `apps/anyscan/public-site.html` now has an enhanced client-side controller layered alongside the older helpers. Entry points to keep aligned are `loadPublicFindingsEnhanced(...)`, `buildPublicFindingsUiQuery(...)`, and the URL/share-link sync helpers. Shared public-search URL params are prefixed (`fq`, `fseverity`, `fdetector`, `fpath`, `flimit`, `fsort`) so they do not collide with other public routes/forms.
- The backend default normalized public findings search limit is currently `20` (`DEFAULT_PUBLIC_FINDINGS_QUERY_LIMIT` in `apps/anyscan/src/core.rs`). The public-site UI still advertises optional larger presets (25/50), but an empty limit field now means the backend default of 20.
- `apps/anyscan/public-site.html` public findings cleanup removed the legacy duplicate helper/loader path. Keep the enhanced flow centered on `loadPublicFindingsEnhanced(...)`, and avoid reintroducing `activePublicFindingsQuery` / `activePublicFindingsResults`; the loader now renders from a local normalized array and the empty limit placeholder is aligned to the backend default of `20`.



- `apps/anyscan` now supports inheriting Dragonfly credentials from the AnyGPT API `.env` while preserving anyscan namespace isolation. The key path is `AppConfig::apply_anygpt_api_redis_credentials(...)` in `src/config.rs`: inherited `REDIS_URL` is normalized to a bare server URL (path stripped to `/`), DB `0` sharing is allowed, and isolation relies on `storage.redis_key_prefix` (default `anyscan:` or an explicit custom prefix).
- Useful targeted verification commands for that storage work: `cd apps/anyscan && cargo test inherited_anygpt_api_credentials_share_db_zero_with_prefix_isolation -- --nocapture && cargo test inherited_anygpt_api_credentials_preserve_explicit_isolated_key_prefix -- --nocapture`, plus ignored live checks `cargo test live_store_inherits_anygpt_api_env_credentials_into_isolated_namespace -- --ignored --nocapture` and `cargo test live_stores_with_distinct_key_prefixes_remain_isolated -- --ignored --nocapture` when Dragonfly credentials are available.
- Direct file formatting in `apps/anyscan` may need `rustfmt --edition 2024 ...`; plain `rustfmt` can mis-detect the edition on `src/bin/anyscan-api.rs` and fail on `async fn` syntax.
- `apps/anyscan/src/bin/anyscan-api.rs` now has focused reviewer-note privacy regressions for both list and SSE sanitization helpers. Useful command: `cd apps/anyscan && cargo test --bin anyscan-api sanitize_ -- --nocapture`.
- Focused end-to-end publication verification can stay lightweight: `cd apps/anyscan && cargo test sanitize_public_finding_moderation_records_hides_private_notes_in_lists && cargo test sanitize_api_event_for_session_hides_public_finding_notes_from_read_only_sessions && cargo test sanitize_api_event_for_session_preserves_public_finding_notes_for_moderators && cargo test public_finding_moderation_controls_ && cargo test inherited_anygpt_api_credentials_share_db_zero_with_prefix_isolation && cargo test inherited_anygpt_api_credentials_preserve_explicit_isolated_key_prefix && cargo test target_request_profile_must_exist && cargo test target_request_profile_is_normalized_when_defined` passed together in this workspace.
- Repo naming has been cut over to `apps/anyscan`, `anyscan-api`, `anyscan-worker`, and `ANYSCAN_*`, but the already-installed live host deployment has not been migrated yet. As of the last live verification before the repo rename, the running system still used `exposure-api`, `/opt/exposure-runtime/bin/exposure-api`, and `/etc/exposure-runtime/runtime.env` with a public bind on `0.0.0.0:8088` and direct Dragonfly/Redis wiring to `10.87.212.41:6380`.
- Important live deployment caveat: the current installed service still runs under the pre-cutover `exposure-api.service` / `exposure-worker.service` layout and service user `exposure-runtime` until the new `apps/anyscan/deploy.sh` is applied. When migrating to the new AnyScan units, keep the old `ProtectHome=true` / `ProtectSystem=strict` behavior in mind: pointing `ANYSCAN_ANYGPT_API_ENV_FILE` at `/root/AnyGPT/apps/api/.env` will still fail unless the file is exposed through a service-accessible path or systemd bind mount.
